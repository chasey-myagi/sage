概述

02

Hook 系统架构

Hook 是挂载在 Agent 生命周期特定事件上的外部副作用。Event 触发 → Runner 并发执行所有匹配 Hook → 根据 exit code 决定是否阻断、注入模型或展示给用户。

Event 触发
Agent 执行到特定生命周期节点（如 PreToolUse、UserPromptSubmit）
Matcher 过滤
按 tool name / if 条件筛选，避免无谓触发
并发执行所有匹配 Hook
stdin 注入 JSON context，同一 event 的所有 hook 并发执行
Exit Code 路由
exit 0 — 成功，stdout 按事件语义决定是否注入模型
exit 2 — 阻断，stderr 注入模型（让模型感知原因）
exit N — 用户可见错误（stderr 展示），执行继续

Hook 配置层级

层级文件作用范围
用户级~/.claude/settings.json所有项目
项目级.claude/settings.json当前项目
Skill 级Skill frontmatter hooks 字段Skill 激活期间自动注册
03

4 种 Hook 类型

从简单的 shell 脚本到启动子 Agent 验证，四种类型覆盖了从轻量回调到复杂验证的全部场景。

类型执行方式适用场景关键字段
command 宿主 shell 执行脚本 80% 场景 — 审计/日志/拦截/注入 command, timeout, async, if
http POST JSON 到 webhook URL 外部系统集成、Slack/飞书通知 url, timeout
prompt 调小模型语义评估 无法用 regex 判断的内容检查 prompt
agent 启动子 Agent 执行验证 需要工具调用的复杂验证逻辑 prompt（作为任务传入子 Agent）

command hook 完整字段

{
  "type": "command",
  "command": "python3 audit.py",
  "if": "Bash(git *)",      // 权限规则语法，细粒度过滤
  "shell": "bash",          // bash | powershell
  "timeout": 30,            // 超时秒数
  "statusMessage": "审计中...",
  "once": false,            // true = 执行一次后移除
  "async": false,           // true = 后台执行，不阻塞
  "asyncRewake": false      // 后台执行时，exit 2 唤醒模型
}

stdin JSON 输入格式

// PreToolUse
{ "tool_name": "Bash", "tool_input": { "command": "rm -rf /tmp" }, "tool_use_id": "..." }

// PostToolUse
{ "tool_name": "Bash", "tool_input": {...}, "tool_use_id": "...", "response": "..." }

// UserPromptSubmit
{ "prompt": "帮我查飞书日程", "session_id": "..." }

// Stop
{ "stop_reason": "end_turn", "session_id": "..." }

27 个 Hook Events

按 Agent 生命周期分组。每个 event 可独立配置 Matcher 和多个 Hook。

会话级 (3)
Session Lifecycle
SessionStart — 会话开始（startup/resume/clear）
SessionEnd — 会话结束
Setup — init/maintenance 触发
用户输入级 (1)
User Input
UserPromptSubmit — 用户发送消息
exit 2 = 阻断 + 清除原始消息
工具调用级 (5)
Tool Execution
PreToolUse — 调用前（exit 2 = 阻断）
PostToolUse — 成功后
PostToolUseFailure — 失败后
PermissionDenied — 权限拒绝
PermissionRequest — 弹出权限框
Agent 结束级 (2)
Agent End
Stop — Agent 结束响应前（exit 2 = 继续对话）
StopFailure — API 错误结束
压缩级 (2)
Compaction
PreCompact — 压缩前（exit 2 = 阻断）
PostCompact — 压缩后
文件系统级 (4)
Filesystem
CwdChanged、FileChanged — 文件变化监听
InstructionsLoaded — CLAUDE.md 加载
ConfigChange — 配置变更（exit 2 = 阻断）
多 Agent 级 (6)
Multi-Agent / Swarm
SubagentStart/Stop、TeammateIdle
TaskCreated/Completed
→ Sage 不需要这组
其他 (4)
Misc
Elicitation/ElicitationResult — MCP 用户输入
WorktreeCreate/Remove — Git worktree
Matcher 机制： 每个 event 可设置 matcher 过滤，避免无谓触发。matcher: "Bash" 只在 bash 工具时触发，if: "Bash(git push*)" 提供更细粒度的条件过滤。
05

Skill 系统

Skill = Markdown 文件 + YAML frontmatter。/skill-name 调用时，完整内容注入当前对话作为 prompt。设计哲学：零编译、用户可读可改、按需加载。

加载来源（优先级从高到低）

policy（企业管控）
最高优先级，企业统一下发
~/.claude/skills/
用户级 Skills，所有项目可用
.claude/skills/
项目级 Skills，当前项目专属
plugin/skills/
插件提供的 Skills（如 superpowers）
bundled（内置 Skill）
编译进二进制，如 /remember，可携带参考文件

Skill Frontmatter 核心字段

---
name: my-skill                    # string，= /command 名
description: "做某件事"            # 展示在 /skills 列表
whenToUse: "当...时调用"           # AI 自动调用的触发描述
allowedTools: [Bash, Read]        # 此 skill 可用的工具
argumentHint: ""            # 参数格式提示（TUI 显示）
model: "claude-haiku-4-5"         # 覆盖默认模型
context: inline                   # inline | fork（fork 开新对话）
userInvocable: true               # false = 仅内部使用

# Skill 可嵌入 Hooks（激活时自动注册）
hooks:
  PostToolUse:
    - matcher: "Write"
      hooks:
        - type: command
          command: "bash .hooks/lint.sh"
---
正文（Markdown）— 注入给 Agent 的 prompt 内容...

懒加载与 Token 预算

启动时（轻量）

  • 只读 frontmatter（name + description + whenToUse）
  • 约 50–100 tokens/skill
  • skill 列表注入 system prompt
  • Agent 知道可以用哪些 skill

调用时（完整加载）

  • /skill-name 触发时加载全文
  • 约 1000–5000 tokens/skill
  • $ARGUMENTS 占位符替换
  • 嵌入的 hooks 自动注册

Bundled Skill 注册方式

registerBundledSkill({
  name: 'remember',
  description: '整理记忆',
  whenToUse: '用户想整理记忆时',
  userInvocable: true,
  files: {
    'SKILL.md': '...',               // 首次调用时解压到临时目录
    'references/foo.md': '...',
  },
  async getPromptForCommand(args, ctx) {
    return [{ type: 'text', text: SKILL_PROMPT }]
  },
})
// 参考文件解压到 ~/.claude/skills-cache///

Sage 设计建议

Sage 是单用户领域专家 Agent 产品，不是多 Agent 协作平台。在 CC 架构基础上精简 + 扩展，适配 Sage 产品形态。

Hook Events — 精选 12 个（去掉多 Agent 协作类，增加 3 个 Sage 特有）

Event优先级来源Sage 特有价值
SessionStartP0CC每次 chat 启动时初始化环境、注入动态上下文
UserPromptSubmitP0CC拦截/增强用户输入（最常用）
PreToolUseP0CC审计工具调用、安全防护（exit 2 = 阻断）
PostToolUseP0CC结果回调、触发副作用
StopP0CCAgent 结束前质量检查，可要求继续
SessionEndP1CC保存状态、清理外部资源
PreCompactP1CC压缩前自定义处理（exit 2 = 阻断）
PostCompactP1CC压缩后通知/记录
FileChangedP1CC监听 workspace 文件变化
MemoryUpdated ★P2Sage 特有MEMORY.md 写入后触发（知识沉淀回调）
AgentSwitched ★P2Sage 特有/switch feishu 切换 Agent 时
WorkspaceChanged ★P2Sage 特有工作目录变更

Sage Hook 配置位置

# ~/.sage/agents/feishu/agent.yaml（Agent 级）
hooks:
  SessionStart:
    - hooks:
        - type: command
          command: "python3 ~/.sage/agents/feishu/init.py"
  PreToolUse:
    - matcher: "bash"
      hooks:
        - type: command
          command: "bash ~/.sage/agents/feishu/hooks/audit.sh"
          if: "bash(curl *)"   # 只在 curl 时触发

# ~/.sage/hooks.yaml（全局，跨所有 Agent）
PreToolUse:
  - matcher: "bash"
    hooks:
      - type: command
        command: "~/.sage/global-hooks/security-audit.sh"

Hook × Skill 联动（最有价值的组合）

---
name: monitor-feishu-doc
description: "持续监听飞书文档变化并自动同步"
hooks:
  FileChanged:
    - matcher: "*.md"
      hooks:
        - type: command
          command: "python3 ~/.sage/agents/feishu/skills/sync-doc.py $ARGUMENTS"
---
# 飞书文档监控模式

激活后，Sage 监听 workspace 中的 .md 文件变化，自动同步到飞书...
联动价值： 激活 /monitor-feishu-doc skill 的同时，FileChanged hook 自动注册。退出会话时自动注销。无需用户手动管理 hook 的生命周期。

结论与实施建议

核心结论

CC 的 Hook + Skill 架构本质上是两个正交的扩展维度：Hook 控制 when（生命周期），Skill 控制 what（专项能力）。两者通过 Skill frontmatter 中的 hooks 字段连接，形成"激活能力同时挂载行为"的强大组合。

Sage 可以完整采用这套设计哲学，只需做两件事：精简 Events（从 27→12，去掉多 Agent 协作类）和增加 per-agent 粒度（agent.yaml 中的 hooks + agents/<name>/skills/ 目录）。
