01

Hook 机制：代码层 Harness

Hook 是挂载在 Agent 生命周期事件上的外部副作用。不依赖 LLM 判断，确定性执行——这是 Hook 与 Prompt 指令的根本区别。LLM 可以犯错，Hook 不会。

为什么 Hook 是独立章节？ Hook 是整个 Sage 知识体系中唯一的"代码层机械保证"。其他四层（Skill / Craft / Wiki / Memory）都依赖 LLM 判断，只有 Hook 在代码层强制执行，构成不可绕过的质量门控和安全拦截线。

三个核心 Hook 事件

PreToolUse — 拦截危险操作
每次工具调用前触发（Layer 2）。exit 2阻断工具调用并将 stderr 注入 Agent 上下文，Agent 看到拦截原因后可调整策略。典型用途：二进制白名单校验、文件路径边界检查、危险命令审计。
Stop — 质量评估（Harness eval 入口）
AgentLoop 完成一轮后触发（Layer 3 退出）。exit 2要求 Agent 继续执行，直到质量标准通过为止。在 Runeforge 场景下，Stop hook 就是 Harness eval 的入口——Runeforge 定义质量标准（eval 脚本），通过 Sage hook 机制注入，这是 Harness 在代码层控制 Sage 行为的那根线。
UserPromptSubmit — 输入校验与增强
收到用户消息、进入 PROCESSING 前触发。在 LLM 看到消息之前处理：权限校验、内容过滤、自动补充上下文、注入结构化指令。

exit code 协议

exit code语义效果
exit 0通过继续正常执行，Hook 无输出
exit 2阻断 + 注入阻断当前操作，将 Hook 的 stderr 注入 Agent 上下文（Agent 可读到原因并重新规划）
exit N（其他）用户可见错误将 stderr 显示给用户，类似工具调用失败

4 种 Hook 类型

主流
command
执行外部命令或脚本（bash / python / 任意二进制）。最常用，灵活度最高。
远程
http
发送 HTTP 请求到外部服务。适合与现有权限系统、审计系统集成。
提示注入
prompt
向 Agent 上下文注入额外系统提示。动态增强 Agent 的上下文感知。
子 Agent
agent
启动另一个 Sage Agent 执行 Hook 逻辑。复杂评估场景（如 LLM-as-judge）。

完整 Hook 事件一览

事件触发时机典型用途
SessionStartDaemonLoop STARTUP 完成后初始化脚本、加载外部数据、环境预检
UserPromptSubmit收到用户消息，进入 PROCESSING 前消息过滤、权限校验、内容增强
PreToolUse每次工具调用前（Layer 2）安全审计、参数拦截（exit 2 = 阻断并注入原因）
PostToolUse每次工具调用后（Layer 2）结果后处理、日志记录、异常检测
StopAgentLoop 完成一轮（Layer 3 退出）质量评估（exit 2 = 要求继续）、Harness eval
PreCompactcontext ≥ 75%，压缩前提示 Agent 整理 MEMORY.md，沉淀知识
PostCompact压缩完成，Memory 重注入后验证注入完整性
SessionEndDaemonLoop SHUTDOWN清理资源、写会话摘要、通知外部系统
FileChangedworkspace 文件变化（Skill 嵌入时注册）监听 CSV/配置变化自动触发同步

配置示例

# agent.yaml — Hook 配置段 hooks: UserPromptSubmit: - type: command command: "python3 hooks/validate_input.py" # exit 2 + stderr → 注入 Agent，说明为什么不处理此消息 PreToolUse: - type: command command: "bash hooks/security_check.sh" # exit 2 → 阻断危险命令，stderr 告知 Agent 原因 Stop: - type: command command: "bash hooks/eval.sh" # exit 0 → 质量通过；exit 2 → 要求 Agent 继续优化 # 在 Runeforge 中：eval.sh 是 Harness 注入的质量评估脚本
全局 Hook vs Agent Hook。 ~/.sage/hooks.yaml 对所有 Agent 生效（安全基线）；agent.yaml 中的 hooks 只对当前 Agent 生效（领域逻辑）。两者合并执行，全局 Hook 先触发。
02

四层知识体系

一个"领域专家 Agent"的可信赖性，依赖四层各自独立、相互协作的知识体系。Skill 保证它是高效的，Craft 保存程序性产物，Wiki 保证它是有知识的，Memory 保证它是懂我的。控制层（Hook）见第一章。

知识层机制解决的问题由谁维护
SkillMarkdown 领域知识（全局只读）领域认知：SOP、最佳实践开发者 / 领域专家
CraftAgent 自管理可复用产物（任意类型）程序性记忆：脚本 / 模板 / SOPAgent 自管理
WikiLLM 维护的结构化知识库领域知识沉淀：概念、模式、陷阱Agent（IDLE 时自动）
Memory操作性短期记忆用户个性化：偏好、决策上下文Agent + 用户

为什么四层缺一不可

没有 Hook
失去确定性保证
Skill / Wiki 依赖 LLM 推理，都可能犯错。只有 Hook 在代码层强制执行——没有 Hook，任何质量标准都只是"建议"。
没有 Skill
效率空白
知道停（Hook），也有领域知识（Wiki），但没有固化的操作流程。每次执行同类任务都要重新推理，token 浪费严重。
没有 Craft
程序性记忆缺失
做过的任务无法沉淀成可复用的脚本、模板、SOP。下次同类任务从零开始，token 效率无法积累提升。
没有 Wiki
知识碎片化
有 SOP（Skill/Craft）但没有领域全貌。会按步骤做，但遇到意外无法举一反三——"会做"但不"理解"。

知识晋升路径

四层知识之间有明确的晋升路径——从即时观察到稳定的领域知识，每层更压缩、更长寿：

messages（Working Memory）
  ↓ compact 前 → 碎片事实写入 memory/
Episodic（raw/sessions/）：完整会话归档，不可变
  ↓ Daemon IDLE 时 wiki-ingest
Semantic（wiki/pages/）：从任务执行中提炼的领域规律
  ↓ 反复出现的模式 → 固化为 Craft
Procedural（craft/）：SOP、脚本、模板、素材，token 效率评分驱动进化

TaskRecord（metrics/）← 并行评测传感器，与晋升路径并行存在

workspace 目录结构

workspace/
├── SCHEMA.md             ← Wiki 锚点：wiki 根路径、frontmatter 规范、[[slug]] 约定
├── AGENT.md              ← Schema 层（顶层认知框架，注入 system prompt）
├── raw/                  ← 第一层：不可变原始记录（Episodic）
│   └── sessions/             每次对话的完整记录，ingest 后标记 processed
├── wiki/                 ← 第二层：LLM 维护的结构化领域知识（Semantic）
│   ├── index.md              所有页面的目录索引（按类别分组）
│   ├── log.md                追加式维护日志（append-only）
│   ├── overview.md           全局综合理解（随知识积累不断演化）
│   └── pages/                所有 wiki 页面，flat 结构，slug 命名
│       ├── feishu-calendar-api.md   ← pitfall 类
│       ├── batch-event-pattern.md   ← pattern 类
│       └── timezone-pitfall.md      ← api-ref 类
├── memory/               ← 操作性短期记忆（Episodic → Semantic 中间层）
│   ├── MEMORY.md             索引文件（注入 system prompt）
│   ├── user.md               用户画像
│   ├── context-*.md          当前项目上下文
│   └── decisions-*.md        关键决策记录
├── craft/                ← Agent 自管理的可复用产物（Procedural Memory）
│   ├── <craft-name>/         SOP 类：CRAFT.md + metrics.json
│   ├── <script>.py           脚本类：可执行脚本
│   └── <template>.md         模板类：输出模板
└── metrics/              ← 任务度量记录（TaskRecord）
    ├── <task_id>.json         每次 UserDriven 任务一条，ULID 命名
    └── summary.json           最近 N 条任务的滚动聚合
03

Wiki 知识沉淀

Wiki 层的设计受 Karpathy 的 LLM-Wiki 模式启发——知识需要分层加工：raw sources → wiki pages → schema，知识在加工过程中产生复利效应。Agent 越用越懂领域，不是每次从零开始。

Sage Wiki ≠ 研究知识库。 kfchou/wiki-skills 的 source 是外部论文/URL（"这篇文章说了什么"）。Sage 的 source 是 Agent 自己的 session records（"做这次任务我学到了什么规律"）——内容是领域操作经验，不是文献摘要。

Daemon IDLE 触发流

UserDriven 会话结束
对话记录归档到 raw/sessions/，标记 unprocessed。Agent 回到 IDLE。
PostProcessing hook 检查
未处理 session 数量 ≥ 阈值？冷却期已过？→ 触发 WikiMaintenance 会话。
WikiMaintenance 会话执行
SessionType 标记为 WikiMaintenance（避免循环触发）。注入 wiki-ingest Skill 到 Agent 上下文，向 Agent 发送自动消息：整理未处理的 session 记录。Agent 使用标准文件工具按 Skill 指令操作 wiki/。
维护完成
标记 sessions 为 processed。若优先级队列中有用户消息，立即处理（WikiMaintenance 期间用户消息入队不中断）。
回到 IDLE
Wiki 知识持续积累。下次对话时 Agent 可通过 wiki-query Skill 检索已沉淀的领域知识。

会话类型：防无限循环

SessionType说明触发 wiki 维护？
UserDriven正常用户对话结束后 → 是
WikiMaintenancewiki 整理会话否（避免循环）
CraftEvaluationCraft 评估 / 创建否

4 个 Wiki Skills（chasey-myagi/llm-wiki）

Wiki 不用自定义 Tool——Agent 用标准文件工具（Read / Write / Glob / Grep），Skill 指令约束行为模式，确保按 llm-wiki 规范操作。

SkillSage 适配后的功能触发时机
wiki-ingest读取 raw session → 提炼领域规律 → 写入 wiki 页面（pitfall / pattern / api-ref / decision）Daemon IDLE 时自动
wiki-query查阅已有领域知识，带 [[slug]] 引用Agent 执行任务需要回忆规律时
wiki-lint健康检查：断链、孤立页、过期页每 5-10 次 ingest 后
wiki-update修订已有页面，展示 diff 后写入发现已有知识需纠正时

Wiki 页面类型

pitfall
踩过的坑
具体报错 + 正确做法。例如："飞书日历 API 批量写入超过 50 条必须分批，否则 rate limit 报错；正确模式：每批 ≤ 20 条，批间 sleep 200ms，重试 3 次"。
pattern
反复有效的操作模式
步骤 + 适用场景。从实际任务中归纳，不是抄文档。
api-ref
领域 API 关键行为
实践发现的关键行为，不是文档抄写。例如时区处理的隐藏陷阱。
decision
重要决策记录
做过的重要决策 + 理由，避免同一问题反复权衡。

为什么用 Skill 而不是 Tool

维度自定义 ToolSkill
实现成本Rust 侧注册多个 Tool，重新编译发版纯 Markdown 文件，零代码
修改灵活性改行为需发版改 Skill 文件即生效
可进化性不变Agent 可在 workspace/craft/ 创建改进版

Skill + Craft 架构

这是两个常被混淆的概念，需要明确区分。Skill 是全局只读的领域知识教科书，Craft 是 Agent 自管理的可复用产物工具箱。

核心区别对比

维度Skill（全局）Craft（workspace）
位置~/.sage/skills/workspace/craft/
内容类型Markdown only任意类型（SOP / 脚本 / 模板 / 素材）
写入者开发者（只读，随平台发版更新）Agent 自管理（craft_manage 工具）
注入方式/skill-name 激活时注入上下文craft_manage 加载时注入
生命周期长期稳定（开发者控制）长期（token 效率越高越稳定）

Craft 类型详解

类型示例格式
SOP / 操作手册飞书日历批量更新步骤CRAFT.md + metrics.json
可执行脚本自动生成周报的 Python 脚本.py / .sh
输出模板日程摘要的 Markdown 格式.md
素材 / 数据生成的报表、配置文件.csv / .json / .pdf

Craft 创建触发条件

Agent 通过 craft_manage 工具自主创建 Craft，触发条件：复杂任务成功完成（5+ 工具调用）、从错误中恢复的有效路径、用户纠正后的正确做法、反复出现的同类任务模式（3+ 次）。

SOP Craft 的 Token 效率评分

仅适用于 SOP 类 Craft（有 CRAFT.md + metrics.json）。评分越高，system prompt 中推荐权重越高——Agent 更倾向使用高效 Craft。

// workspace/craft/batch-calendar-update/metrics.json
{
  "craft_name": "batch-calendar-update",
  "type": "sop",
  "executions": [
    { "ts": "2026-04-14T10:00:00Z", "task_hash": "abc123", "tokens_used": 1200, "success": true },
    { "ts": "2026-04-15T09:00:00Z", "task_hash": "abc123", "tokens_used": 980,  "success": true }
  ],
  "avg_tokens": 1090,
  "best_tokens": 980,
  "score": 0.90,        // score = best_tokens / avg_tokens，越接近 1 越高效稳定
  "version": 3
}
score = best_tokens / avg_tokens。 越接近 1.0，说明每次执行都接近最优路径——Craft 非常稳定高效。Score 低的 Craft 触发 CraftEvaluation 会话，Agent 自动重写优化。System prompt 中按 score 降序排列，引导 Agent 优先使用高效 Craft。

评测系统：TaskRecord

四层知识体系的评测传感器——没有它，Skill 的 efficiency_score 是空的，数据驱动优化无从谈起。MetricsCollector 订阅已有的 AgentEvent 流，零侵入 agent_loop，UserDriven 会话结束时自动写入。

采集事件采集内容
TurnEnd.message.usage每轮 input / output / cache token 消耗（逐轮累加）
ToolExecutionEnd工具调用次数、失败次数
CompactionEnd压缩次数
AgentEnd / RunError任务成功 / 失败 + 原因
数据积累后可以回答的问题： 这个 Skill 是否真的省了 token？这套 model 配置性价比如何？同类任务第 N 次是否比第 1 次便宜了？本质是用经验数据做提示蒸馏——不是梯度下降，是观察 → 沉淀 → 提炼 → 优化的闭环。

Daemon 架构：第 4 层循环

Sage Agent 是 Daemon 常驻进程，不是会话式应用。用户通过 sage connect 接入，断开后 Agent 继续运行等待下一个消息。Agent 不需要知道"什么时候结束"——它不结束，只是等待。

4 个命令

sage start feishu          → 启动 feishu daemon（加载 Memory，Agent 待命）
sage connect feishu        → 接入 TUI，和 Agent 对话（可多次 connect/disconnect）
sage disconnect            → 断开 TUI（Agent 继续运行，等待下次连接）
sage stop feishu           → 关闭 daemon，资源释放

sage send feishu "查下明天日程"   → 非交互式，发一条消息，等结果返回
sage status                        → 列出所有运行中的 Agent daemon

DaemonLoop 状态机

DaemonLoop 是 Agent ReAct 三层循环之上的第 4 层循环（代码层，不可 YAML 配置）：

第 4 层 DaemonLoop
  STARTUP: 加载 AgentConfig → 注入 Memory → 加载 Skills → 注册 Hooks
  loop {
    IDLE:
      → PostProcessing hook：有未处理 session 记录？
        → 有：进入 WIKI_MAINTENANCE
        → 无：rx.recv().await（挂起等待用户消息，让出 tokio 调度权）
    PROCESSING（SessionType::UserDriven）:
      fire UserPromptSubmit hook
      第 3 层 OUTER LOOP（follow-up 队列）
        第 2 层 INNER LOOP（steering + tool calls）
          第 1 层 INNERMOST（LLM 流式调用）
          fire PreToolUse hook → tool 执行 → fire PostToolUse hook
      fire Stop hook（exit 2 = 要求继续）
      归档对话记录到 raw/sessions/（标记 unprocessed）
    WIKI_MAINTENANCE（SessionType::WikiMaintenance）:
      注入 wiki-ingest Skill，向 Agent 发送自动消息
      Agent 使用标准工具操作 wiki/，完成后标记 sessions 为 processed
      ※ 此期间用户新消息进入优先级队列，不中断
    回到 IDLE
  }
  SHUTDOWN: fire SessionEnd hook

VM Pool：LRU 资源调度

DaemonLoop 是轻量 async task（可以很多个），microVM 是重量级资源（受 RAM 限制）。VM Pool 用 LRU 算法弥合两者之间的阻抗差。

IDLE_HOT
IDLE_COLD
PROCESSING
BLOCKED
状态VM有活跃用户响应延迟
IDLE_HOT运行中否毫秒（即时）
IDLE_COLD无（已驱逐）否1-2s（cold start）
PROCESSING运行中是—（正在执行）
BLOCKED等待分配是（有消息）等待空槽
资源效率示例： M=3 个 VM 槽，N=10 个已注册 Agent。同时活跃 ≤ 3 个：零等待，即时响应。第 4 个被唤醒：LRU 驱逐最久未用的，~1s cold start。大多数时候总 RAM 占用 = 3 个 VM，不是 10 个。

与 CC / Codex 的本质差异

维度Claude Code / CodexSage

workspace可切换（不同项目）固定（每 Agent 一个专属空间）

安全模型

Sage 面对的核心威胁：LLM 生成的代码不可信，任务可能从远程派发，多 Agent 共享同一台机器。两种沙箱模式对应不同场景。

两种沙箱模式

模式适用场景工具来源安全性
none（本地默认）本地部署、可信环境、早期用户宿主机 PATH（brew / npm / pip 工具全部可用）无沙箱，类 Claude Code
microvm（生产升级）服务器、多租户、多用户共享rootfs + tools/ 持久化卷（Linux 二进制）硬件虚拟化隔离

4 层纵深防御（microvm 模式）

Layer 0Hardware Virtualization（KVM / HVF）
隔离内存、CPU、中断。突破条件：VM escape 漏洞（极罕见）。
Layer 1microVM（msb_krun）
最小 rootfs（sage-guest + busybox）、独立 Linux 内核（libkrunfw）、Volume mount 只挂载显式允许的路径。网络策略：airgapped / TSI 白名单。
Layer 2Guest Agent 策略执行
二进制白名单（ToolPolicy.allowed_binaries）、文件路径约束（allowed_read/write）。exec 前拦截，不在白名单直接拒绝。
Layer 3Landlock LSM + seccomp-bpf
内核级系统调用白名单（seccomp），文件系统访问范围约束（Landlock）。即使 Layer 2 被绕过，内核层仍拦截。
